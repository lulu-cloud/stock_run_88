## 2. 项目亮点怎么讲

### 2.1 亮点一：不是聊天机器人，而是闭环 Agent 系统

面试表达：

> 这个项目不是简单把 LLM 接到一个聊天框，而是把 Agent 放到一个可执行的业务闭环里。交易 Agent 每天读取行情、持仓、订单、记忆和技能，调用工具生成条件单，系统撮合后再生成报告和复盘，再把复盘结果压缩成记忆注入下一次决策。

关键词：

- 感知
- 决策
- 执行
- 反馈
- 记忆
- 进化

### 2.2 亮点二：Agent 自我进化可控

面试表达：

> 我没有让记忆无限增长，而是做了固定大小的分层记忆。trade_fact 存客观规律，trade_prefer 存主观偏好，short_ring 存近 3 日极端行情，交易体系文档做长期抽象。每次注入提示词前都会压缩，避免上下文污染和 token 爆炸。

关键词：

- bounded memory
- memory compression
- reflection
- skill confidence
- prompt injection control

### 2.3 亮点三：赛马机制不越权

面试表达：

> 一开始我设计过根据盈亏强制限制 Agent 仓位，后来调整为只做评价和提示。因为如果目标是模拟真实交易员，系统不应该因为短期亏损直接剥夺交易权限，而应该把赛马指标作为复盘信号交给 Agent 自主解释。代码层只保留合规和硬规则，比如现金、T+1、涨跌停、板块准入。

这体现了：

- Agent 自主性。
- 业务规则边界。
- 评价系统和执行系统解耦。

### 2.4 亮点四：推荐助手复用交易员知识

面试表达：

> 股票推荐助手不是孤立推荐，而是读取交易 Agent 的公开交易体系、赛马表现和高置信技能，再结合用户画像生成推荐。这样推荐逻辑可以溯源到真实交易员的复盘和实盘表现。

关键词：

- cross-agent memory sharing
- recommendation trace
- user feedback
- trading-backed recommendation

## 3. 面试官可能问的项目问题与回答

### Q1：你这个 Agent 和普通 LLM Chat 有什么区别？

回答：

普通 LLM Chat 主要是一次性问答，而这个系统的 Agent 有完整生命周期：

1. 每日读取结构化交易上下文。
2. 调用工具获取行情、政策、公司业务、订单历史。
3. 输出结构化交易计划。
4. 订单进入撮合系统。
5. 交易结果进入日报和复盘。
6. 复盘结果更新记忆和技能置信度。
7. 下一次决策前注入压缩后的进化上下文。

所以它是一个带工具、状态、记忆、执行反馈和评估的 Agent 系统。

### Q2：为什么要做工具白名单？

回答：

工具白名单有三个作用：

1. 控制 Agent 行为边界，避免所有 Agent 调用同一批工具导致策略同质化。
2. 降低 token 和工具调用成本。
3. 让前端可以把 Agent 配置成不同角色，例如短线情绪 Agent 可以多用板块和 K 线工具，基本面 Agent 可以多用公司业务和政策工具。

工具白名单不是提示词建议，而是实际传给 LLM binding 的工具集合。

### Q3：为什么赛马机制最后不强制限制仓位？

回答：

因为这个系统的目标是模拟交易员，而不是基金风控系统。真实交易员亏钱后会被考核和提醒，但不一定有程序直接限制他下一单的仓位。  

所以我把赛马机制定位为评价系统：

- 计算收益、回撤、夏普、胜率等指标。
- 生成风格标签。
- 作为提示词注入和复盘依据。

代码层只保留硬规则：

- 现金不足不能买。
- T+1 不满足不能卖。
- 涨跌停价格非法不能入库。
- 未授权板块不能买。

这是评价系统和执行约束的边界设计。

### Q4：你的自我进化具体体现在哪里？

回答：

主要有四层：

1. 记忆进化：日终复盘把交易结果、失败订单、市场场景写入 `trade_fact`、`trade_prefer`、`short_ring`。
2. 技能进化：根据近期交易和失败率调整技能置信度。
3. 体系反思：周度、月度或事件触发交易体系文档，抽象出投资哲学、选股标准、风控边界和能力圈。
4. 提示词注入：次日决策前读取压缩记忆、技能索引和交易体系摘要，作为 Agent 的上下文。

### Q5：如何避免记忆越积越多？

回答：

我做了 bounded memory：

- 每类记忆都有字符上限。
- 超限时优先调用 LLM 压缩重写。
- 压缩失败时走确定性裁剪。
- 压缩目标是保留高信号、可执行、已验证的规则，删除流水账和重复判断。

这能避免 token 爆炸和旧结论污染。

### Q6：你的交易 Agent 如何处理工具失败？

回答：

工具调用外层有异常捕获，失败会返回错误文本给 LLM。  

另外对关键环节还有后端兜底：

- LLM 输出 JSON 解析失败时返回空计划。
- 挂单价不合法时触发修复回合。
- 最终入库前再次做价格校验。
- 买单冻结资金前检查现金。
- 卖单检查可卖数量和 T+1。
- 未成交订单会过期释放冻结资金。

也就是说，LLM 可以犯错，但入库和撮合层要兜住。

### Q7：为什么需要交易体系文档？

回答：

日常记忆更像经验片段，交易体系文档是更高层抽象。  

比如一次失败订单只说明“某只股票没成交”，但多次失败后需要抽象为：

- 该 Agent 不适合追高开过大的票。
- 某类行情下挂单价格过保守。
- 某个技能在退潮期失效。

交易体系文档把零散经验升级成可复用方法论，并且有版本记录。

### Q8：推荐助手和交易 Agent 怎么协作？

回答：

推荐助手读取交易 Agent 的公开体系摘要、赛马指标和高置信技能，但不暴露未成交条件单。  

当用户问股票推荐时，助手可以引用表现较好的交易员体系，生成有实战背书的推荐逻辑。用户反馈会回流到推荐技能置信度和推荐助手记忆。

### Q9：你如何评估推荐助手？

回答：

目前已有用户反馈和推荐 trace。后续会扩展三类评估：

- 用户反馈：点赞、点踩、太激进、逻辑清晰、采纳率。
- 后验行情：推荐后 T+1/T+3/T+5 收益，是否跑赢大盘和板块。
- 工程成本：响应时间、token 消耗、工具调用次数、失败率。

这样可以同时评估推荐准确性、用户满意度和系统成本。

### Q10：如果引入公司财报，你会怎么做？

回答：

我会新增一个财报解读 Agent，而不是把所有逻辑塞进交易 Agent。  

财报 Agent 负责：

- 解析营收、利润、毛利率、现金流、负债、存货、应收账款。
- 输出结构化评分。
- 标记红旗风险。
- 给出财报是否支持当前市场叙事。

交易 Agent 只消费结构化结果，这样可以降低上下文复杂度，也方便单独评估财报 Agent。

## 4. 常见 Agent 八股问题与回答

### Q1：什么是 Agent？

回答：

Agent 是能在目标驱动下感知环境、规划动作、调用工具、观察结果并迭代决策的系统。  

它和普通 LLM 的区别是：

- 有目标。
- 有工具。
- 有状态。
- 有记忆。
- 有反馈。
- 能多步执行。

### Q2：ReAct 是什么？

回答：

ReAct 是 Reasoning + Acting。LLM 不是一次性输出答案，而是在推理和行动之间循环：

```text
Thought -> Action(tool call) -> Observation -> Thought -> ...
```

在项目中，交易 Agent 会先判断需要什么信息，再调用行情、政策、选股、订单历史、价格计算等工具，最后输出交易计划。

### Q3：Function Calling 和 Agent 的关系是什么？

回答：

Function Calling 是 Agent 的动作接口。  

Agent 负责规划和选择工具，Function Calling 负责把自然语言意图转成结构化函数调用。  

只有 Function Calling 不等于 Agent，因为 Agent 还需要：

- 任务目标。
- 多步循环。
- 结果观察。
- 状态管理。
- 记忆与评估。

### Q4：Agent 设计最重要的是什么？

回答：

我认为最重要的是边界：

- 哪些事情交给 LLM 判断。
- 哪些事情必须由代码规则兜底。
- 哪些工具允许调用。
- 记忆怎么压缩。
- 失败如何恢复。
- 如何评估 Agent 是否真的变好。

项目里我把交易判断交给 Agent，把现金、T+1、涨跌停、板块准入这些硬规则放在代码层。

### Q5：Agent 记忆有哪些类型？

回答：

常见分为：

- 短期记忆：当前对话或当前任务上下文。
- 长期记忆：跨任务保留的事实、偏好、经验。
- 情景记忆：具体事件和执行记录。
- 语义记忆：抽象规则和知识。
- 反思记忆：从结果中总结出来的方法论。

项目中对应：

- `short_ring`：短期行情记忆。
- `trade_fact`：客观规律。
- `trade_prefer`：主观偏好。
- `system/current.md`：反思后的交易体系。

### Q6：RAG 和 Agent 有什么关系？

回答：

RAG 是给 LLM 提供外部知识的方式，Agent 是能规划和行动的系统。  

Agent 可以把 RAG 当成工具之一。  

例如：

- 政策文档检索。
- 公司主营业务检索。
- 财报检索。
- 历史交易体系检索。

RAG 解决“知道什么”，Agent 解决“接下来做什么”。

### Q7：RAG 常见问题有哪些？

回答：

常见问题：

- 文档切分不合理。
- 召回不准。
- chunk 缺上下文。
- embedding 不适配领域。
- 没有 rerank。
- 引用来源缺失。
- 检索内容过多导致噪声。
- LLM 对检索结果幻觉补全。

解决思路：

- 结构化解析文档。
- 元数据过滤。
- hybrid search。
- rerank。
- query rewrite。
- 引用来源。
- answer grounding。
- 对召回质量单独评估。

### Q8：如何评估 Agent？

回答：

不能只看最终回答，要分层评估：

- 任务成功率。
- 工具调用正确率。
- 工具调用必要性。
- JSON schema 合规率。
- 失败恢复能力。
- 延迟。
- token 成本。
- 用户反馈。
- 业务后验指标。

在交易项目中，业务后验就是收益、回撤、成交率、订单失败率；推荐助手后验就是推荐后 T+1/T+3/T+5 表现和用户反馈。

### Q9：如何降低 Agent token 成本？

回答：

做法：

- 工具结果截断。
- 只注入必要上下文。
- 记忆固定大小压缩。
- 技能索引渐进加载。
- 工具白名单。
- 避免机械调用所有工具。
- 将长文档先结构化摘要。
- 对重复上下文做缓存。

项目中已经做了记忆压缩、工具结果截断、工具白名单和技能索引。

### Q10：如何防止 Agent 幻觉？

回答：

不能完全靠提示词，需要工程约束：

- 关键数据必须来自工具。
- 输出前做 schema 校验。
- 价格、现金、权限由代码层校验。
- 要求引用数据来源。
- 对工具结果做结构化返回。
- 对 LLM 输出做后处理和二次修复。

项目里挂单价必须调用计算工具和校验工具，最终入库前还会做后端校验。

## 5. 面试时可主动展开的系统设计

### 5.1 如果面试官问“怎么扩展到多专家 Agent？”

回答结构：

1. 先拆角色：
   - 政策 Agent
   - 基本面 Agent
   - 财报 Agent
   - 技术面 Agent
   - 风控 Agent
   - 组合经理 Agent
2. 每个专家输出结构化 JSON。
3. 组合经理做冲突合并和优先级排序。
4. 交易 Agent 只消费压缩后的专家结论。
5. 每个专家单独评估。

可以画流程：

```text
政策/基本面/财报/技术面 Agent
        -> 组合经理 Agent
        -> 交易 Agent
        -> 订单系统
        -> 复盘与评价
```

### 5.2 如果面试官问“怎么做评估系统？”

回答结构：

交易 Agent 评估：

- 收益率。
- 超额收益。
- 回撤。
- 夏普。
- 胜率。
- 盈亏比。
- 成交率。
- 订单过期率。
- token 成本。
- 决策耗时。
- 工具失败率。

推荐助手评估：

- 推荐后收益。
- 用户点赞/点踩。
- 用户采纳率。
- 风险提示完整度。
- 推荐 trace 完整度。
- 响应时延。
- token 成本。

### 5.3 如果面试官问“为什么不用单 Agent？”

回答：

单 Agent 的问题是上下文过长、职责混乱、评估困难。  

多 Agent 可以：

- 降低单个 Agent 上下文压力。
- 让每个 Agent 专注一个领域。
- 单独评估专家能力。
- 复用专家结论。
- 降低提示词复杂度。

但多 Agent 也有成本：

- 调度复杂。
- token 成本更高。
- 专家冲突需要合并。
- 需要更强的观测和评估。

所以我会先在关键高价值链路引入专家 Agent，例如政策和财报。

## 6. 简历中可以量化的指标

如果你后续补数据，可以写得更有冲击力：

- 支持 N 个交易 Agent 并行赛马。
- 支持 N 类 Function Calling 工具。
- 每日自动生成 N 类报告/日志。
- 订单撮合覆盖买入、卖出、冻结、过期、开盘成交。
- 记忆压缩控制在固定字符上限内。
- Telegram 推荐助手支持用户画像、反馈和 trace。

如果暂时没有真实统计，不要编数字。可以写“设计并实现”而不是“提升 xx%”。

## 7. 面试表达模板

可以用这段作为开场：

> 我最近做了一个 A 股多 Agent 智能投顾系统，重点不是预测股票，而是把 Agent 工程闭环做完整。系统里有交易 Agent 和 Telegram 股票推荐助手。交易 Agent 每天读取行情、持仓、订单、进化记忆和技能置信度，调用工具完成选股、行情分析、价格计算和订单校验，然后生成条件单。后端负责撮合、资金冻结、T+1、板块权限等硬规则。每天收盘后系统会做复盘，更新记忆、技能置信度和交易体系文档，第二天再注入提示词。我还做了赛马评价机制和前端 Agent 配置看板，可以配置风格提示词、工具白名单、优先策略和板块权限。

## 8. 需要避免的说法

不要说：

- “这个系统能稳定赚钱。”
- “Agent 可以自动预测股票涨跌。”
- “完全实现了自我进化。”
- “推荐一定准确。”

建议说：

- “这是一个模拟交易和 Agent 工程验证系统。”
- “重点是闭环、评估、记忆和工具调用。”
- “交易结果用于验证假设，而不是承诺收益。”
- “自我进化目前是记忆、技能置信度和体系反思层面的工程实现，后续会引入更严格统计验证。”

## 9. 后续补强方向

求职前建议优先补：

1. Agent 评估看板：
   - token。
   - 耗时。
   - 工具调用次数。
   - 工具失败率。
2. 推荐助手后验收益：
   - T+1/T+3/T+5。
   - 跑赢大盘/板块。
3. 多专家 Agent 原型：
   - 政策 Agent。
   - 基本面 Agent。
   - 财报 Agent。
4. 提示词预览页面：
   - 面试展示很直观。
5. 一页架构图：
   - 数据层。
   - 工具层。
   - Agent 层。
   - 执行层。
   - 评估/进化层。
