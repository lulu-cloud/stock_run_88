# A股多Agent智能投顾系统功能梳理与后续路线图

更新时间：2026-05-17

## 1. 项目定位

这是一个面向 A 股模拟交易与股票推荐场景的多 Agent 系统。核心目标不是只做一个“会调用工具的聊天机器人”，而是把交易 Agent、股票推荐助手、数据流水线、订单撮合、Telegram 推送、前端配置和自我进化机制串成完整闭环。

当前系统主要包含两类 Agent：

- 交易 Agent：面向每日复盘、条件单生成、订单撮合、持仓管理和交易体系进化。
- 股票推荐助手 Agent：面向 Telegram 用户问答、股票推荐、用户画像、推荐逻辑溯源和反馈学习。

## 2. 已实现功能

### 2.1 交易 Agent 基础能力

已支持多个交易 Agent 并行运行：

- 追高打板 Agent：偏短线情绪、连板、高开抢入、主线加速。
- 自主决策 Agent：偏政策、基本面、技术、资金、情绪多因子综合。
- 自定义 Agent：可通过前端配置风格提示词、工具白名单、优先策略和阶段提示词。

交易 Agent 每日复盘时会读取：

- 当前可用资金、冻结资金、总资产、持仓市值、浮盈浮亏。
- 当前持仓。
- 上一交易日成交。
- 近期挂单、失败/过期原因。
- 进化记忆快照。
- 技能置信度。
- 交易体系文档摘要。
- Agent 专属配置：风格提示词、工具白名单、优先选股策略、板块权限、阶段提示词。

Agent 输出结构化 JSON：

```json
{
  "market_analysis": "大盘分析",
  "selected_stocks": [],
  "orders": [],
  "risk_assessment": "风险评估"
}
```

订单支持字段：

- `open_get_in`：集合竞价/开盘抢入或抢出标志。
- `reserved_cash`：买单冻结资金。
- `skill_id`：下单所用技能。
- `skill_confidence`：下单时技能置信度。
- `evolution_mark`：进化标签。
- `fail_reason`：失败/过期原因。
- `failure_attribution`：失败归因。

### 2.2 订单与撮合

当前订单链路支持：

- 买单冻结资金。
- 卖单校验可卖股数和 T+1。
- 隔日未成交 pending 单自动过期并释放冻结资金。
- 新复盘生成次日预操作单时替换旧预操作单。
- 开盘激进成交：
  - 买单：`open_get_in=true` 且开盘价小于等于限价，则按开盘价成交。
  - 卖单：`open_get_in=true` 且开盘价大于等于限价，则按开盘价成交。
  - 开盘不满足则继续走普通限价撮合。
- 普通限价撮合：当日区间触达限价则成交。

已取消“赛马亏损强制限制仓位”的代码规则。赛马只做评价和提示，不强制修改 Agent 仓位、不自动暂停、不自动取消订单。

### 2.3 股票板块准入规则

当前唯一保留的代码级交易权限规则是“板块准入”。

默认自动模式：

- 主板/中小板：始终允许。
- 创业板：总资产大于等于 20 万，且距离第一笔真实成交满 60 自然日后自动解锁。
- 科创板/北交所：总资产大于等于 60 万，且距离第一笔真实成交满 60 自然日后自动解锁。

前端支持切换为手动模式：

- 手动模式下以前端开关为准。
- 买入未授权板块会被代码层拦截。
- 卖出已有持仓不受板块准入限制。

### 2.5 赛马机制

赛马机制现在是评价系统，不再是强制仓位控制系统。

核心指标：

- `race_score`：综合评分。
- `excess_return`：Agent 当日收益率减上证指数收益率。
- `max_drawdown`：最大回撤。
- `sharpe_ratio`：收益波动比。
- `calmar_ratio`：累计收益 / 最大回撤。
- `win_rate`：卖出交易胜率。
- `profit_factor`：盈利总额 / 亏损总额。
- `beta_score`：收益与大盘同向程度的近似代理。
- `alpha_score`：累计收益扣除 Beta 代理后的表现。
- `style_tag`：风格标签，例如情绪进攻型、Alpha稳健型、贝塔依赖型、高回撤型。

赛马结果用于：

- 前端竞技场展示。
- Agent 详情页说明。
- 每日提示词中作为复盘和风险提示。
- 交易体系反思输入。

赛马结果不用于：

- 不强制限制总仓位。
- 不强制限制单票仓位。
- 不自动暂停 Agent。
- 不自动取消 pending 订单。

### 2.7 Telegram 与股票推荐助手

Telegram 侧已经具备：

- Bot 状态查询。
- Chat 绑定 Agent。
- 日报推送。
- 推送模块全局开关。
- 用户画像。
- 关注股。
- 自然语言推荐入口。
- 推荐 trace。
- 推荐反馈。
- 推荐助手记忆文件。
- 推荐技能置信度调整。

股票推荐助手当前更像“轻量 Agent 化推荐器”：

- 能读取交易员公开体系文档。
- 能读取赛马指标和高置信技能。
- 能结合用户画像做推荐表达。
- 能记录用户反馈并调整推荐技能置信度。
- 有推荐助手自身的压缩记忆文件。

但它还不是完整的多轮规划型 Agent：

- 推荐链路仍以规则 + 工具 + LLM 文案为主。
- 还没有完整 ReAct 工具循环。
- 推荐技能库还比较粗。
- 用户反馈与后验股票走势还没有形成严格评价闭环。

## 3. 每日流程

### 3.1 数据阶段

每日数据更新流程：

1. 调度器检查是否到行情拉取时间。
2. 拉取上证指数和 A 股日线数据。
3. 检查 CSV 数据新鲜度。
4. 数据不足时进入退避重试。
5. 多次失败后跳过本轮并推送等待数据通知。

### 3.2 撮合阶段

对每个启用的交易 Agent：

1. 自动取消上一交易日仍未成交 pending 单。
2. 释放未成交买单冻结资金。
3. 按最新行情更新持仓市值。
4. 撮合当天 pending 单。
5. 再次更新持仓市值和账户资产。

### 3.3 决策阶段

构造 AgentContext：

- 现金与资产。
- 持仓。
- 成交记录。
- 近期订单。
- 冻结资金。
- 进化上下文。
- Agent 配置。

LLM Agent 执行 ReAct 工具循环：

1. 读取大盘、政策、板块。
2. 调用策略选股。
3. 分析个股 K 线、业务、政策相关性。
4. 查看近期失败订单。
5. 计算挂单价格。
6. 校验挂单价。
7. 输出结构化交易计划。

### 3.4 入库阶段

1. 校验挂单价是否在参考价涨跌停范围内。
2. 校验买入板块权限。
3. 买单冻结资金。
4. 卖单校验可卖数量和 T+1。
5. 条件单写入 `agent_order`。

### 3.5 报告与进化阶段

1. 生成 `report.md`。
2. 写入 `agent_daily_report`。
3. 执行日终进化：
   - 分钟级复盘。
   - 技能置信度更新。
   - 记忆更新。
   - 风控/因子日志写入。
4. 计算赛马指标。
5. 触发周度/月度/事件型交易体系反思。
6. 到推送时间后 Telegram 推送日报。

## 4. Hermes 式自我进化体现

当前系统的 Hermes 式自我进化主要体现在五个层面。

### 4.1 分层记忆

每个交易 Agent 有独立记忆目录：

```text
agent_memory/{agent_id}/
├── trade_fact.md
├── trade_prefer.md
├── short_ring.md
├── snapshots/
└── system/
    ├── current.md
    └── versions/
```

含义：

- `trade_fact.md`：客观规律，只保留已验证市场事实。
- `trade_prefer.md`：主观偏好，记录 Agent 风控偏好和避坑规则。
- `short_ring.md`：近 3 日极端行情短期缓冲。
- `snapshots/`：每日决策前冻结记忆快照。
- `system/current.md`：交易体系文档。
- `system/versions/`：交易体系版本。

### 4.2 记忆固定大小与自动压缩

记忆不能无限增长，因为每日会注入提示词。

当前代码约束：

- `trade_fact.md`：固定上限。
- `trade_prefer.md`：固定上限。
- `short_ring.md`：固定上限。
- `system/current.md`：固定上限。
- 推荐助手 `recommend_memory.md`：固定上限。

超限时会触发压缩：

- 优先用 LLM 重写成高密度 Markdown。
- 失败时使用确定性裁剪兜底。
- 删除流水账、重复句和空泛判断。

### 4.3 技能置信度

每个 Agent 有技能库：

- `momentum_hunt`
- `risk_compression`
- `balanced_factor`
- `switch_sequence`

技能记录：

- 市场场景。
- 当前置信度。
- 最近失败率。
- 动态参数。
- 失效场景。
- 进化记录。

日终复盘会根据近期交易和失败订单更新技能置信度。

### 4.4 分钟级复盘

系统只对 Agent 操作过的股票拉取/构造分钟级复盘输入，避免全市场分钟线带来的成本浪费。

主要用于：

- 判断非原子换仓风险。
- 分析先卖 A 再买 B 是否在时间顺序上可行。
- 为失败订单归因提供盘中结构。
- 给自我进化提供更细粒度证据。

### 4.5 交易体系反思

周期触发：

- 最近 5 个交易日完成后触发周度反思。
- 月末窗口触发月度反思。
- 近期失败率过高触发事件反思。

输出：

- 核心投资哲学。
- 选股与择时标准。
- 风控与仓位管理。
- 能力圈边界。

该文档会被压缩后注入后续决策提示词，形成：

```text
交易行为 -> 结果验证 -> 归因 -> 记忆更新 -> 技能置信度更新 -> 体系文档更新 -> 次日提示词注入
```

### 4.6 股票推荐助手的轻量自我进化

推荐助手已有：

- 推荐技能表。
- 用户画像。
- 用户反馈。
- 推荐 trace。
- 推荐助手记忆文件。
- 反馈后调整推荐技能置信度。

当前闭环：

```text
用户提问 -> 推荐生成 -> 引用交易员体系 -> 用户反馈 -> 更新推荐技能置信度 -> 写入推荐助手记忆
```

下一步需要把后验走势也纳入推荐评估。

## 5. 当前架构的不足

### 5.1 交易 Agent 不足

- Agent 的交易逻辑仍主要依赖 LLM 推理，缺少 ML/DL 因子工具辅助。
- 技能参数更新还比较简单，更多是启发式调整。
- 分钟级复盘还没有形成稳定的可视化回放。
- 交易体系文档更新缺少统计显著性验证。
- 工具调用成本、耗时和成功率尚未纳入 Agent 评分。

### 5.2 股票推荐助手不足

- 推荐助手还没有完整 ReAct loop。
- 推荐后验评估不足，尚未自动跟踪推荐标的 T+1、T+3、T+5 表现。
- 用户反馈维度还偏粗。
- 推荐技能库不够细。
- 对交易员体系的引用还偏摘要，缺少可解释证据链。

### 5.3 多 Agent 协作不足

当前多 Agent 更像“多交易员并行赛马”，还不是完整的专家协作团队。

可拆分更多专门 Agent：

- 政策解读 Agent。
- 公司基本面 Agent。
- 财报解读 Agent。
- 产业链 Agent。
- 技术面 Agent。
- 风控审查 Agent。
- 组合经理 Agent。
- 推荐话术 Agent。

### 6.2 多 Agent 架构演进

建议从“并行交易 Agent”升级为“专家协作 Agent Team”。

可设计如下角色：

#### 政策解读 Agent

职责：

- 读取发改委、工信部、财政部等政策。
- 抽取行业、主题、强度、时间敏感度。
- 判断政策是否对持仓或候选股构成催化。

输出：

```json
{
  "industries": [],
  "policy_strength": 0.0,
  "benefit_stocks": [],
  "risk_stocks": [],
  "evidence": []
}
```

#### 公司基本面 Agent

职责：

- 搜索/读取公司主营业务。
- 梳理收入结构、核心产品、行业位置。
- 判断公司是否真实受益于主题。

输出：

```json
{
  "business_summary": "",
  "theme_relevance": 0.0,
  "key_products": [],
  "risk_points": []
}
```

#### 财报解读 Agent

职责：

- 读取季报/年报。
- 解析营收、利润、毛利率、现金流、负债、存货、应收账款。
- 判断财报是否支持当前股价叙事。

输出：

```json
{
  "financial_quality": 0.0,
  "growth_score": 0.0,
  "cashflow_score": 0.0,
  "red_flags": []
}
```

#### 技术面 Agent

职责：

- 负责 K 线、量价、均线、MACD、涨停结构。
- 输出买卖点和失效条件。

#### 风控 Agent

职责：

- 审查交易员订单。
- 不基于盈亏强制限制仓位。
- 只做合规和硬规则检查：
  - 现金是否足够。
  - T+1 是否满足。
  - 板块权限是否满足。
  - 涨跌停价格是否合法。

#### 组合经理 Agent

职责：

- 汇总各专家意见。
- 分配最终候选优先级。
- 协调多个交易 Agent 的观点差异。
- 输出最终给交易 Agent 或推荐助手的上下文。

建议演进流程：

```text
政策 Agent -> 基本面 Agent -> 财报 Agent -> 技术面 Agent
             \             |             /
              \            |            /
               -> 组合经理 Agent -> 交易 Agent / 推荐助手
```

### 6.3 交易员评估系统

建议新增 `agent_eval_metric` 表。

指标维度：

- 收益：
  - 日收益率。

  - 最大回撤。

- 交易质量：
  - 胜率。
  - 盈亏比。
  - 平均持仓天数。
  - 换手率。
  - 订单成交率。
  - pending 过期率。
  - 开盘抢入成功率。
- Agent 工程指标：
  - token 消耗。
  - 工具调用次数。
  - 工具调用失败率。
  - LLM 响应时间。
  - 决策总耗时。

## 7. 可对外讲的工程亮点

1. 不是单一 LLM Chat，而是完整多 Agent 交易闭环。
2. 有真实工程流水线：数据、撮合、订单、冻结资金、报告、推送、前端。
3. 有 Agent 工具系统，支持工具白名单和策略参数化调用。
4. 有 Hermes 式自我进化：记忆、技能、反思、压缩、版本。
5. 有赛马评价机制，但保留真实交易员自主性，不用代码强制因亏损限仓。
6. 有股票推荐助手，与交易员体系打通。
7. 有前端可配置 Agent 工厂雏形。
8. 有明确可扩展方向：ML/DL function call、多专家 Agent、评估系统。
